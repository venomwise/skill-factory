# Spec Exec

Implement tasks from a `tasks.md` file (generated by the spec-plan skill), one at a time, validating each against the acceptance criteria in the companion `requirements.md`. Track progress by updating checkboxes in `tasks.md` after each task completes.

The core loop is simple: pick the next incomplete task, implement it, validate it, mark it done, repeat. The details below handle edge cases — checkpoints, optional tasks, failures, and resumption.

## When to use

- Implementing tasks from a spec-plan-generated `tasks.md`
- Resuming execution of a partially completed plan
- Checking progress on an existing plan

## When NOT to use

- There's no `tasks.md` or it has no `## Tasks` section
- You just need a quick single-file edit unrelated to any plan

## Locating the files

The user may provide the path directly, or you may need to find it. Common locations:

- Path provided by the user (highest priority)
- `specs/` or `docs/specs/` directories
- `.codex/specs/` directory
- Current directory

Look for a `tasks.md` that contains a `## Tasks` section. The companion `requirements.md` is usually in the same directory (linked from the Overview section of tasks.md).

If multiple specs exist, list them and ask the user which one to execute.

## Understanding the tasks.md format

```
- [ ] Phase 1: Title
  - [ ] 1.1 Task title
    - Description of what to do
    - _Requirements: 1.1, 1.2_
  - [ ] 1.2 Another task [optional]
    - Description
    - _Requirements: 1.3_

- [ ] Checkpoint: Verify <what to validate>

- [ ] Phase 2: Title
  ...
```

Key elements:
- **Phase lines**: `- [ ] Phase N: Title` — group related tasks
- **Task lines**: `- [ ] N.M Task title` — the actual work items
- **Description lines**: indented bullets under a task — what to do, which files to touch
- **Requirement references**: `_Requirements: N.M_` — acceptance criteria IDs from requirements.md
- **Optional tasks**: task title ends with `[optional]` — can be skipped in MVP mode
- **Checkpoints**: `- [ ] Checkpoint: ...` — pause points for user validation
- **Completed tasks**: `- [x] N.M Task title` — already done

## Workflow

### 1. Load and assess

Read `tasks.md` and `requirements.md`. Scan progress:
- Count completed (`[x]`) and remaining (`[ ]`) tasks
- Identify the first incomplete task
- If all tasks are complete, report the status and stop

Present a brief summary to the user: "X of Y tasks complete. Next up: task N.M — <title>."

### 2. Handle optional tasks

If the plan contains tasks marked `[optional]`, ask the user once at the start:

> "Some tasks are marked as optional (tests, nice-to-haves). Would you like to: A) Skip optional tasks (MVP mode) B) Execute all tasks"

Remember the choice for the rest of the session. In MVP mode, skip any task whose title contains `[optional]`.

### 3. Execute the task loop

For each task in `## Tasks` order:

**Skip** if:
- Already marked `[x]`
- Marked `[optional]` and MVP mode is active

**Checkpoint** — if the task title contains "Checkpoint":
- Summarize what's been completed so far
- Run any verification steps described in the checkpoint
- Ask the user to confirm before continuing
- Mark as `[x]` after confirmation

**Implement** — for regular tasks:

a. Read the description lines and `_Requirements:` references under the task
b. Look up the referenced requirement IDs in `requirements.md` to understand the acceptance criteria
c. Review the files/modules mentioned in the description before making changes — understand existing code first
d. Implement the task following project conventions
e. Validate:
   - Run any explicit validation steps from the task description
   - Check each referenced acceptance criterion is satisfied
   - If tests exist and are relevant, run them
f. Mark completion — this is critical, do it immediately:
   - Change `- [ ]` to `- [x]` for the completed task
   - Write the change to `tasks.md` NOW, before starting the next task
   - Never batch multiple completions into one write

**Mark phase completion** — mark a Phase line as `[x]` only when every sub-task under it is `[x]`. If any sub-task is still `[ ]` — even if it was skipped because it's optional in MVP mode — leave the Phase as `[ ]`. The Phase checkbox reflects actual completion state, not "nothing left to do".

### 4. Handle failures

If a task fails or gets blocked:
- Keep it as `- [ ]` (do not mark it done)
- Explain what went wrong
- Ask the user: "Should I try to fix this, skip it and continue, or stop here?"
- If the failed task produces artifacts needed by later tasks, warn about potential cascading issues

### 5. Report completion

When all tasks are done (or all non-optional tasks in MVP mode):
- Summarize what was implemented
- Note any tasks that were skipped or failed
- Suggest next steps if applicable (e.g., "You may want to run the full test suite")

## Rules

- Update `tasks.md` after EVERY completed task — one task, one write. Progress must survive interruptions.
- Never alter task numbering, titles, or descriptions — only change checkboxes.
- Only modify items under the `## Tasks` section.
- Read referenced files before modifying them — understand before changing.
- Match the user's language in all communication.
- If execution is blocked or unclear, stop and ask rather than guessing.
