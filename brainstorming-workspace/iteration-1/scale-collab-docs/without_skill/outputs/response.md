# 支持十万人同时在线的协同文档系统设计

## 概述

要构建一个支持十万人同时在线的协同文档系统，核心挑战在于：**实时同步**、**冲突解决**、**水平扩展**和**低延迟**。以下从架构设计、技术选型和关键难点三个维度给出建议。

---

## 一、架构设计

### 1. 整体架构分层

```
客户端（浏览器/App）
      ↓ WebSocket / HTTP
  接入层（负载均衡 + 网关）
      ↓
  实时协同服务（Collab Service）
      ↓
  文档存储层（操作日志 + 快照）
      ↓
  消息队列 + 发布订阅
```

### 2. 接入层

- 使用 **Nginx / Envoy** 做 L7 负载均衡，将 WebSocket 连接分散到多个节点。
- 每个接入节点维护一定数量的长连接（单台机器支撑 1~5 万并发连接是合理目标）。
- 部署 3~10 台接入节点即可覆盖十万并发。

### 3. 实时协同服务

- 每个文档的编辑会话由一个**房间（Room）**管理，同一文档的所有用户归属同一 Room。
- Room 可以通过 **一致性哈希** 分配到不同的服务实例，避免单点瓶颈。
- 服务实例之间通过 **Redis Pub/Sub** 或 **Kafka** 做跨节点消息转发。

### 4. 数据存储层

- **操作日志（Operation Log）**：每次编辑产生一条操作记录，持久化到数据库（如 PostgreSQL 或 TiDB）。
- **文档快照（Snapshot）**：定期（如每 100 次操作或每 5 分钟）生成完整快照，避免重放日志过长。
- **缓存层**：用 Redis 缓存热点文档的当前状态，减少数据库读压力。

## 二、核心技术：冲突解决

多人同时编辑最难的问题是**操作冲突**。主流方案有两种：

### 方案 A：OT（Operational Transformation，操作变换）
- Google Docs 采用的方案。
- 服务端有一个**权威版本**，所有操作先发到服务端，服务端变换后广播。
- 优点：逻辑相对成熟，易于实现权限控制。
- 缺点：中心化，服务端是瓶颈；实现复杂（尤其是富文本场景）。

### 方案 B：CRDT（Conflict-free Replicated Data Type，无冲突复制数据类型）
- Figma、Notion 等新一代产品逐步采用。
- 数据结构本身保证任意顺序合并都能得到一致结果，不需要中心化仲裁。
- 推荐库：**Yjs**（性能极好，社区活跃）、**Automerge**。
- 优点：天然支持离线编辑、P2P 场景；扩展性更好。
- 缺点：数据结构较复杂，存储体积略大。

**推荐选择 Yjs + WebSocket**，这是目前开源生态中最成熟、性能最好的协同方案。

## 三、技术选型建议

| 层次 | 推荐技术 |
|------|---------|
| 前端框架 | React / Vue + Tiptap / ProseMirror（富文本编辑器） |
| 协同算法 | Yjs（CRDT） |
| 实时传输 | WebSocket（主）+ HTTP Long Polling（降级） |
| 接入层 | Nginx + Kubernetes Ingress |
| 协同服务 | Node.js / Go（高并发长连接） |
| 跨节点同步 | Redis Pub/Sub 或 Kafka |
| 持久化存储 | PostgreSQL（操作日志）+ S3（文档快照/附件） |
| 缓存 | Redis Cluster |
| 监控 | Prometheus + Grafana |
| 容器编排 | Kubernetes（水平自动扩缩容） |

## 四、关键难点与应对策略

### 1. 连接风暴（大量用户同时上线）
- 接入层使用连接限速 + 队列机制，避免瞬间冲垮服务。
- WebSocket 握手与业务逻辑分离，接入层只做路由。

### 2. 热点文档（同一文档百人以上同时编辑）
- 对超大并发文档，可以采用**分片广播**：将用户分组，减少单次广播扇出。
- 设置单文档在线人数上限（如 500 人），超出后进入
