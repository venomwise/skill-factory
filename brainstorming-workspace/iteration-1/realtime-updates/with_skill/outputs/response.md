# 为现有 REST API 添加实时推送能力

## 问题诊断：Solution Unclear

你的目标很清晰：**数据变更时，客户端能立即收到更新**，不靠轮询。现有 REST API 已经存在，需要在其基础上叠加推送能力。技术路径是开放的，所以我们直接进入方案对比。

在此之前，有一个假设值得确认：

> **假设**：主要客户端是浏览器或移动端 App（而非其他后端服务）。如果消费方主要是后端服务，消息队列（Kafka / RabbitMQ）是更自然的选择，方案会完全不同。

以下三个方案均假设客户端是前端 / 移动端。

---

## 三种主流方案对比

### 方案 A（推荐）：Server-Sent Events（SSE）

**核心思路**：在现有 HTTP 服务上新增一个 `GET /events` 端点，服务端持有长连接，数据变更时主动向客户端推文本流。

| 维度 | 评估 |
|------|------|
| 协议 | 标准 HTTP，无需握手升级 |
| 通信方向 | 单向（服务端 → 客户端） |
| 客户端支持 | 浏览器原生 `EventSource` API；移动端有成熟库 |
| 基础设施改动 | 最小；nginx/ALB 需允许长连接 |
| 断线重连 | 浏览器自动重连；`Last-Event-ID` 支持断点续传 |
| 适合场景 | 通知推送、Feed 更新、仪表盘实时数据 |
| 主要限制 | 纯单向；HTTP/1.1 每域名 6 连接上限（HTTP/2 无此问题） |

**为什么首推 SSE**：复用现有 HTTP 栈，不引入新基础设施，运维成本最低，风格与 REST API 最一致。对于"客户端只需接收更新"这个目标，SSE 是最小化方案。

---

### 方案 B：WebSocket

**核心思路**：通过 HTTP Upgrade 握手建立全双工 TCP 连接，客户端和服务端均可主动发消息。

| 维度 | 评估 |
|------|------|
| 协议 | WS / WSS，需握手升级 |
| 通信方向 | 全双工（双向） |
| 客户端支持 | 浏览器原生；移动端广泛支持 |
| 基础设施改动 | 中等；反向代理需支持 WS 升级；水平扩展需 sticky session 或共享状态（Redis Pub/Sub） |
| 断线重连 | 需自行实现或用库（socket.io 等） |
| 适合场景 | 聊天、协作编辑、游戏、客户端也需向服务端推消息的场景 |
| 主要限制 | 引入连接状态管理复杂度；与 REST 的无状态哲学不同 |

**选这个的条件**：客户端需要双向通信（如用户在客户端触发操作需要低延迟反馈），或产品路线图中有实时协作功能。

---

### 方案 C：长轮询（Long Polling）

**核心思路**：客户端发 HTTP 请求，服务端挂起直到有新数据或超时，客户端收到响应后立即再次请求。

| 维度 | 评估 |
|------|------|
| 协议 | 标准 HTTP |
| 通信方向 | 单向（模拟推送） |
| 基础设施改动 | 几乎为零 |
| 实时性 | 接近实时，但有一次 RTT 的延迟 |
| 适合场景 | 需要兼容极老旧环境，或临时过渡方案 |
| 主要限制 | 服务端并发连接压力大；延迟高于 SSE/WebSocket；现代项目不推荐作为长期方案 |

**选这个的条件**：只作为短期兼容方案，或目标环境不支持 SSE/WebSocket（极少见）。

---

## 推荐方案：SSE + 内部事件总线

基于"最小改动、最快落地"原则，推荐 **方案 A（SSE）** 配合一个轻量内部事件总线：

```
数据变更
  └→ 业务逻辑层触发事件（内存 EventEmitter 或 Redis Pub/Sub）
        └→ SSE 连接池：找到订阅该资源的连接
              └→ 向客户端推送 text/event-stream 消息
```

单节点部署用内存 EventEmitter 即可；多节点水平扩展时换成 Redis Pub/Sub，其余代码不变。

## 需要你确认的关键问题

在开始写设计文档之前，我需要了解以下几点（逐一回答即可）：

**问题 1（最重要）**：客户端需要订阅哪种粒度的更新？

- (A) 全局广播：所有登录用户都收到同一条更新（如系统公告）
- (B) 按资源订阅：客户端订阅特定资源（如 `GET /events?resource=order&id=123`），只收到该资源的变更
- (C) 按用户隔离：每个用户只收到属于自己的数据更新
- (D) 以上多种混合

这个选择直接影响连接管理和事件路由的设计复杂度。

## 一个值得提前想清楚的盲点

**连接状态与数据一致性**：客户端断线重连期间，数据变更不会丢失吗？

SSE 的 `Last-Event-ID` 可以让客户端告诉服务端"我上次收到的是事件 #42"，服务端可以补发缺失的事件。但这要求服务端有事件持久化（哪怕是短时间的 Redis 列表或数据库记录）。

如果断线期间的变更可以丢失（客户端重连后拉一次完整状态即可），则不需要这个机制，方案简单很多。
