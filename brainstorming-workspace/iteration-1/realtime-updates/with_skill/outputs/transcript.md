用户请求：为现有 REST API 添加实时推送能力，数据变更时客户端能立即收到更新。

### Step 3: 探索项目上下文

任务说明为通用项目（无具体代码库），跳过文件系统探索。上下文设定：存在一个现有 REST API，技术栈未知，需叠加实时推送能力。

### Step 4: 模糊度诊断

对照诊断表评估用户请求：
- 问题清晰：客户端要立即收到数据变更
- 目标清晰：不靠轮询，推送驱动
- 边界基本清晰：在现有 REST API 上扩展
- **技术路径开放**：SSE / WebSocket / Long Polling 均可行

**诊断结论：Solution Unclear**，按技能指引跳到 Step 4（方案对比），不需要先做发散探索。

### Step 5: 方案对比与推荐（Converge and Propose）

识别出三个主流方案并进行对比：

1. **SSE（推荐）**：复用 HTTP 栈，单向推送，基础设施改动最小，与 REST 风格一致
2. **WebSocket**：全双工，适合双向通信场景，但引入状态管理复杂度
3. **长轮询**：零基础设施改动，但延迟较高，仅适合兼容或过渡场景

推荐 SSE + 内部事件总线（单节点用 EventEmitter，多节点换 Redis Pub/Sub）。

### Step 6: 关键问题与盲点

按技能要求，单次只问最重要的一个问题：
- **关键问题**：订阅粒度（全局广播 / 按资源 / 按用户 / 混合）——直接决定连接管理和路由设计复杂度

按盲点扫描清单（集成类），识别出最重要的一个盲点：
- **断线重连期间的数据一致性**：`Last-Event-ID` 可解决但需事件持久化；若允许丢失则方案大幅简化

### Step 7: 输出

生成 `response.md`，包含：
- 模糊度诊断说明
- 前置假设确认
- 三方案对比表格
- 推荐方案架构草图
- 单个关键问题（多选）
- 单个盲点提示

## 关键设计决策记录

- 首推 SSE 而非 WebSocket：用户需求是单向推送，WebSocket 的双向能力在此场景是过度设计
- 事件总线抽象：让单节点和多节点部署路径一致，降低后期扩容成本
- 只问一个问题：遵循技能 "one question at a time" 原则，选择影响最大的订阅粒度问题
