# 分析维度参考

PRD 评审的分析粒度原则：**聚焦业务逻辑闭环和架构合理性，不纠结实现细节。**

## ✅ 应该质疑的（逻辑层面）

- **业务流程闭环：**
  - 登录失败后用户能否重试？重试次数有限制吗？账号锁定后如何解锁？
  - 支付失败后订单状态如何变化？如何触发退款？退款失败怎么办？
  - 审批流程中，如果审批人离职了怎么办？能否撤回？能否转审？

- **边界和异常：**
  - 并发情况下库存扣减会不会超卖？
  - 用户注销后历史数据如何处理？关联的订单、评论怎么办？
  - 超过最大文件大小、最大数量限制时如何提示和处理？
  - 网络超时、第三方服务不可用时的降级方案是什么？

- **状态一致性：**
  - 分布式场景下如何保证状态一致？
  - 异步任务失败后如何回滚或重试？
  - 跨系统数据同步延迟时用户看到的是什么状态？

- **权限和安全：**
  - 谁有权限执行这个操作？权限如何校验？
  - 是否涉及敏感数据？如何脱敏？数据保留多久？
  - 是否有 SQL 注入、XSS、CSRF 等安全风险？

- **性能和规模：**
  - 预期的并发量、数据量是多少？
  - 是否需要分页、缓存、异步处理？
  - 是否有性能瓶颈（如全表扫描、N+1 查询）？

## ❌ 不应该质疑的（实现细节）

这些属于开发阶段确定的实现细节，PRD 评审不关注：

- 错误提示用 toast 还是 modal？用什么文案？
- 字段叫 `user_id` 还是 `userId`？
- 按钮放在左边还是右边？用什么颜色、icon？
- 用 Redis 还是 Memcached？用 REST 还是 GraphQL？
- 代码怎么组织？用什么设计模式？

## 三个结构化分析维度

按以下三个维度组织评审产出：

**1. 逻辑漏洞与边界缺失**
- 正常流程 + 异常分支是否完整？
- 边界条件（空值、极值、并发、重试、超时）是否考虑？
- 状态机是否有死锁或孤岛状态？
- 多角色、多场景下的交互逻辑是否闭环？

**2. 技术可行性与架构风险**
- 现有架构能否支持？改造成本如何？是否需要引入新组件？
- 性能要求是否现实？（基于预期的并发量、数据量评估）
- 安全/合规要求是否可达？（如 GDPR、等保、行业规范）
- 第三方依赖的稳定性和成本如何？有无备选方案？
- 是否与现有功能冲突或重复？

**3. 需 PM 澄清的疑问（大方向）**
- 量化指标缺失：响应时间要求？并发量？数据量级？保留时长？
- 业务规则模糊：优先级如何判断？去重规则是什么？谁有权限？
- 交互逻辑不明确：失败后用户能做什么？成功后跳转到哪里？
- 前置条件缺失：依赖哪些数据已存在？依赖哪些系统已完成？
