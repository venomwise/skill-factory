## 1. 逻辑漏洞与边界缺失

### [P0-阻断] 优惠券库存与发放总量未定义，并发领取存在超发风险
- **问题描述：** PRD §2.2「领取规则」仅说明「用户点击领取即可获得」，未定义每种优惠券的发行总量、单用户领取上限、每日发放上限。在高并发活动场景下，多个领取请求同时通过「是否还有库存」校验会导致超发（实发数量超过预算）。
- **潜在影响：** 优惠券发放数量不可控，直接造成营销预算超支与资金损失；活动无法按预算收口。
- **引用：** PRD §2.2 领取规则

### [P0-阻断] 优惠券抵扣金额计算规则缺失，可能出现负数订单或抵扣异常
- **问题描述：** PRD §2.3 称「优惠券金额从订单总额中扣除」，但：① 满减券「满 100 减 20」的「满 100」判定基准未定义（是商品原价、实付价，还是含/不含运费）；② 折扣券「8 折」是否设置封顶金额未说明；③ 当抵扣金额 ≥ 订单金额时（如订单 90 元用满 100 减 20 不满足、或折后金额为 0/负）订单金额如何处理未定义。
- **潜在影响：** 抵扣计算口径不一致会导致少扣/多扣、出现 0 元或负数订单、对账困难，引发资损和客诉。
- **引用：** PRD §2.1 优惠券类型、§2.3 使用规则

### [P0-阻断] 优惠券与订单的全生命周期状态机未定义（核销、退款、取消未闭环）
- **问题描述：** PRD 仅描述「领取」和「下单抵扣」两个动作，缺失关键状态流转：① 下单时优惠券是「锁定」还是「直接核销」？② 用户取消订单 / 支付失败 / 申请退款后，已使用的优惠券是否退回？退回后有效期如何计算？③ 优惠券状态机（未使用 → 锁定 → 已使用 → 已退回 / 已过期）完全缺失。
- **潜在影响：** 缺少状态闭环会导致优惠券被错误占用或重复使用（一券多用），退款场景下用户权益受损或平台资损。
- **引用：** PRD §2.3 使用规则、§4 验收标准

### [P1-需确认] 优惠券使用门槛与适用范围未定义
- **问题描述：** PRD 未说明优惠券的使用门槛与适用范围：满减券是否有最低订单门槛（除「满 100」外）、是否限定适用商品 / 品类 / 店铺、是否排除特定商品（如已打折商品、虚拟商品）。折扣券是否有最低消费门槛同样未提及。
- **潜在影响：** 适用范围不清会导致优惠券被用于不期望的场景，营销目标无法实现，且后续补充规则需返工。
- **引用：** PRD §2.1、§2.3

### [P1-需确认] 多张优惠券叠加 / 互斥规则未定义
- **问题描述：** PRD §2.3 称「下单时选择一张优惠券进行抵扣」，但未说明：满减券与折扣券能否叠加使用、与平台其他促销（如限时折扣、新人券）是否互斥、同一订单是否仅限一张券。「选择一张」是产品规则约束还是仅当前版本限制，需明确。
- **潜在影响：** 叠加规则不清会出现折上折等超预期优惠组合，侵蚀利润；行业内此处是最常见的资损与滥用点。
- **引用：** PRD §2.3 使用规则

### [P1-需确认] 优惠券有效期定义模糊，过期处理不明确
- **问题描述：** PRD §2.2 称「每张优惠券有有效期，过期后自动失效」，但有效期是固定截止日期、还是领取后 N 天动态计算，未定义。「自动失效」的触发方式（定时任务批量置失效，还是使用时实时判断）及失效后是否通知用户，均未说明。
- **潜在影响：** 有效期计算口径不一致会导致用户在临界点能否使用产生争议；批量失效若依赖定时任务，存在延迟导致过期券仍可用的窗口。
- **引用：** PRD §2.2 领取规则

### [P2-优化] 领取与下单的异常分支（重复点击、网络超时）未描述
- **问题描述：** PRD 未描述用户重复点击「领取」、领取请求超时、下单时优惠券恰好过期 / 被占用等异常分支下的用户提示与系统行为。
- **潜在影响：** 异常分支缺失虽不阻塞主流程，但会影响体验并可能因重试导致重复领取，建议在设计阶段补齐。
- **引用：** PRD §2.2、§2.3

## 2. 技术可行性与架构风险

> 说明：当前为绿地项目，无既有架构可验证。以下风险基于优惠券系统行业通用实践提出，需在技术选型时结合实际方案复核。

### [P0-阻断] 领券防超发依赖原子操作，普通「先查后扣」实现不可行
- **技术分析：** 若按 PRD 字面「点击即领」用「查询剩余库存 → 判断 → 扣减」的非原子实现，在并发下必然超发（read-then-write 竞态）。行业标准做法是用 Redis Lua 脚本原子扣减 + DB 唯一索引兜底幂等，或分布式锁保证单用户领取数量校验准确。
- **改造成本：** 需引入缓存层（如 Redis）做库存扣减与限领计数、消息队列做异步落库与对账，绿地项目需从零搭建该链路。
- **改进建议：**
  - 方案 A：Redis Lua 原子扣减库存 + 用户领取记录唯一索引兜底（优点：高并发性能好，业界成熟；缺点：需引入缓存与一致性保障）。
  - 方案 B：仅数据库行锁 + 唯一索引（优点：实现简单、强一致；缺点：高并发下锁竞争成为瓶颈，仅适合低并发活动）。
  - 具体选型由架构团队决定，本报告不替代技术选型。
- **引用：** PRD §2.2；行业实践见文末来源

### [P1-需确认] 抵扣金额计算与下单流程的事务一致性需保障
- **技术分析：** 「优惠券锁定 → 创建订单 → 支付 → 核销」涉及多步操作，需保证：优惠券核销与订单创建的原子性（避免券已核销但订单创建失败，或订单已成但券未核销）、支付失败 / 超时后的回滚。行业实践建议按 saga（预留 → 应用 → 扣款 → 确认，失败逆向补偿）实现。
- **改造成本：** 需设计优惠券「锁定 / 释放 / 核销」的事务边界与补偿机制。
- **改进建议：** 建议下单时先「锁定」券、支付成功后再「核销」、支付失败 / 取消时「释放」回退；具体由设计阶段确定。
- **引用：** PRD §2.3；行业实践见文末来源

### [P1-需确认] 量级与性能指标缺失，无法评估架构选型
- **技术分析：** PRD 未给出预期用户量、活动峰值领取 QPS、优惠券发行总量、下单峰值等量化指标。是「小活动几千张」还是「大促百万张抢券」，对应的架构方案（单机 DB vs Redis+MQ 集群）截然不同。
- **改造成本：** 取决于量级，差异可达数量级。
- **改进建议：** 需 PM / 运营提供量级预估后才能评估架构可行性与成本（见 Todo List）。
- **引用：** PRD §1 背景、§2.2

## 3. 需与 PM 二次确认的疑问清单 (Todo List)

> 以下问题可直接转发给 PM，无需二次编辑。

- [ ] **关于发放总量与限领：** 每种优惠券的发行总量是多少？单个用户领取上限是几张？是否有每日发放上限？预期活动峰值领取 QPS 大约多少？
- [ ] **关于抵扣计算基准：** 满减券「满 100」的判定基准是商品原价、实付价还是含运费？折扣券「8 折」是否设置最高优惠封顶金额？折后金额是否包含运费？
- [ ] **关于抵扣边界：** 当订单金额恰好等于或小于优惠券面额时（如折后为 0 或负），订单金额如何处理（置 0、不允许使用、还是按门槛拦截）？
- [ ] **关于优惠券状态闭环：** 下单选用优惠券时是「锁定」还是「立即核销」？用户取消订单 / 支付失败 / 退款时，优惠券是否退回？退回后有效期如何计算？
- [ ] **关于使用门槛与适用范围：** 折扣券是否有最低消费门槛？两类券是否限定适用商品 / 品类 / 店铺？是否排除已打折商品或虚拟商品？
- [ ] **关于叠加与互斥：** 满减券与折扣券能否在同一订单叠加？与平台其他促销活动是否互斥？「一单一券」是产品硬规则还是当前版本的临时限制？
- [ ] **关于有效期：** 有效期是固定截止日期还是「领取后 N 天」动态计算？过期失效是定时批量处理还是使用时实时判断？过期前是否需要通知用户？
- [ ] **关于异常处理：** 用户重复点击「领取」或下单时券恰好过期 / 被占用时，期望的提示与系统行为是什么？
