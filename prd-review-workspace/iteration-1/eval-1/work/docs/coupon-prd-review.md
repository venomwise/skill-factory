## 1. 逻辑漏洞与边界缺失

### [P0-阻断] 满减/折扣的"门槛"与"抵扣"计算基准未定义
- **问题描述：** PRD §2.1 定义"满 100 减 20""8 折"，§2.3 仅说"优惠券金额从订单总额中扣除"。但未定义：①"满 100"的判定基准是商品原价小计、还是已扣其他优惠后的金额？②折扣券是对整单打折还是对单品打折？③折扣券是否有封顶金额（如"8 折最多减 50"）？④抵扣后运费、税费是否参与计算？
- **潜在影响：** 计算基准不同会直接导致结算金额错误，引发资损或用户投诉；折扣无封顶在大额订单下可能造成超预期补贴。
- **引用：** PRD §2.1 优惠券类型、§2.3 使用规则

### [P0-阻断] 领取并发与超领/超发控制缺失
- **问题描述：** PRD §2.2 仅说"点击领取即可获得"。未定义：①优惠券是否有发放总量上限（库存）？②单用户对同一券的领取张数上限？③用户快速重复点击/接口重试时如何保证幂等（不会一次操作领到多张）？
- **潜在影响：** 行业内此类系统最典型的资损点。无库存上限与幂等控制，在并发或重试下会出现超发、超领，直接造成营销预算失控与资金损失。
- **引用：** PRD §2.2 领取规则

### [P0-阻断] 下单使用优惠券的并发占用与状态机未定义
- **问题描述：** PRD §2.3 说"下单时选择一张优惠券抵扣"，但未定义优惠券的完整状态机。一张券至少涉及：未使用 → 锁定（下单未支付）→ 已使用（支付成功）→ 已退回（下单取消/支付失败/超时）。PRD 完全没有"下单后未支付""支付失败""订单取消"时优惠券如何回退的描述。
- **潜在影响：** 若下单即标记"已使用"，订单取消后券不退回 → 用户损失券、客诉；若下单不锁定，则同一张券可被并发提交到多个订单重复抵扣 → 资损。这是必须在需求层面闭环的状态机。
- **引用：** PRD §2.3 使用规则、§4 验收标准

### [P1-需确认] 优惠券有效期的边界规则不明确
- **问题描述：** PRD §2.2 说"每张优惠券有有效期，过期后自动失效"，但未定义：①有效期是固定截止日，还是"领取后 N 天有效"？②有效期判定以下单时刻还是支付时刻为准（在临界时间下单、过期后才支付如何处理）？③已锁定在未支付订单上的券，若在支付前过期，该订单还能否完成？
- **潜在影响：** 临界时间判定不一致会导致用户"看似可用却抵扣失败"或反之，产生争议。
- **引用：** PRD §2.2 领取规则

### [P1-需确认] 多优惠券叠加与互斥规则缺失
- **问题描述：** PRD §2.3 说"选择一张优惠券"，可推断为单券，但未明确：①满减券与折扣券能否叠加？②与平台其他营销活动（如限时折扣、新人价）能否叠加？③多商品订单中，券作用于整单还是可指定商品？
- **潜在影响：** 叠加/互斥规则是优惠券系统最常见的"被薅羊毛"漏洞来源；规则缺失会被用户找到组合套利路径。
- **引用：** PRD §2.3 使用规则

### [P1-需确认] 退款/退货时优惠券与退款金额的处理逻辑缺失
- **问题描述：** PRD 全文未涉及售后场景。当使用了优惠券的订单发生退款时：①退给用户的是实付金额还是订单原价？②优惠券是否退回？③部分退款（多商品退其一）时优惠如何按比例分摊？
- **潜在影响：** 若退实付却又退回券，等于用户白赚优惠；若退原价则平台多退钱。这是直接的资损/客诉点。
- **引用：** PRD §4 验收标准（仅覆盖下单抵扣，无售后）

### [P2-优化] 最低实付金额与"减为负/减为零"边界未定义
- **问题描述：** 满减券面额可能接近或超过订单金额（如订单 100 元用满 100 减 20 后是 80；但若存在"满 50 减 50"类券，或订单恰好等于门槛），需明确抵扣后实付是否允许为 0、是否设最低实付金额（如至少 0.01 元以触发支付/对账）。
- **潜在影响：** 实付为 0 可能导致支付通道异常或对账困难。属边界完善项。
- **引用：** PRD §2.3 使用规则

## 2. 技术可行性与架构风险

### [P0-阻断] 缺少量化指标，无法评估架构与性能可行性
- **技术分析：** 当前为空白工程，无既有架构。PRD 未给出任何规模指标：预期发放量级、领取峰值 QPS、是否为"秒杀式"集中领取、参与活动的用户量级。优惠券领取在行业内是典型高并发热点场景——是否需要 Redis 预扣库存 + Lua 原子扣减 + 异步落库，完全取决于这些指标。
- **改造成本：** 若为低频小流量，单库事务即可；若为大促秒杀级，需引入缓存预扣、分布式锁/幂等、库存分片、异步发券队列，成本量级差异巨大。
- **改进建议：**
  - 方案 A（低并发）：MySQL 行锁 + 唯一索引保证幂等，实现简单。适用于发放量小、非集中领取场景。
  - 方案 B（高并发）：Redis 预扣 + Lua 原子校验（库存/限领）+ 消息队列异步落库 + 领取记录分片。适用于秒杀式大促。
  - 选型须由 PM 提供的量级指标驱动，本报告不代替做技术选型。
- **引用：** PRD 全文（无任何量化指标）；行业实践见末尾参考来源

### [P1-需确认] 防刷与风控要求未提及
- **技术分析：** PRD 未提及任何防作弊要求。优惠券是高价值营销资源，常见攻击包括：脚本批量注册领券、单设备多账号薅羊毛、机器抢券。需求层面需明确是否要做设备指纹、领取频控、实名/账号资质校验。
- **改造成本：** 风控规则若后置补做，往往需改动领取链路与数据模型，代价高于一开始预留。
- **改进建议：** 建议在需求阶段即明确风控强度（频控/设备维度/账号资质），以便设计阶段预留扩展点。
- **引用：** PRD §2.2 领取规则

### [P1-需确认] 营销预算与对账机制未定义
- **技术分析：** 优惠券直接对应真金白银的营销支出，但 PRD 未提及预算控制（活动总预算/单券成本统计）与财务对账（优惠金额如何记账、与订单/支付对账）。
- **改造成本：** 对账与预算统计若需求缺失，后期补做需回溯历史数据，成本高。
- **改进建议：** 明确是否需要预算上限熔断、优惠金额的财务记账口径，并在数据模型中预留优惠流水。
- **引用：** PRD §1 背景、§4 验收标准

## 3. 需与 PM 二次确认的疑问清单 (Todo List)

> 以下问题可直接转发给 PM，无需二次编辑。

- [ ] **关于 优惠门槛计算基准：** "满 100 减 20"中的"满 100"，是按商品原价小计判定，还是按已扣除其他优惠后的金额判定？是否包含运费/税费？
- [ ] **关于 折扣券规则：** "8 折"是对整单打折还是对单品打折？是否设封顶金额（如最多减 X 元）？
- [ ] **关于 发放总量：** 每种优惠券是否有发放总量上限（库存）？达到上限后领取入口如何表现？
- [ ] **关于 单用户领取限制：** 同一用户对同一种券最多可领几张？是否限制每人每活动总张数？
- [ ] **关于 领取幂等：** 用户快速重复点击或客户端重试时，期望"只发一张"还是"按次数发放"？
- [ ] **关于 使用并发与状态机：** 下单选用优惠券后、支付完成前，券处于什么状态？订单取消/支付失败/支付超时时，券是否自动退回用户？退回后有效期如何计算？
- [ ] **关于 有效期判定：** 有效期是固定截止日还是"领取后 N 天"？以下单时刻还是支付时刻判定是否过期？
- [ ] **关于 叠加与互斥：** 一个订单是否只能用一张券？满减券与折扣券能否叠加？能否与平台其他促销（限时折扣/新人价等）叠加？
- [ ] **关于 售后退款：** 使用了优惠券的订单退款时，退实付金额还是订单原价？券是否退回用户？部分退款时优惠如何分摊？
- [ ] **关于 规模指标：** 预期发放量级、参与用户量级、领取峰值 QPS 分别是多少？是否为大促"秒杀式"集中领取？（直接决定技术方案选型）
- [ ] **关于 防刷风控：** 是否需要领取频控、设备维度去重、账号资质校验等防薅羊毛能力？强度要求如何？
- [ ] **关于 预算与对账：** 是否需要活动总预算上限/熔断？优惠金额的财务记账与对账口径是什么？
