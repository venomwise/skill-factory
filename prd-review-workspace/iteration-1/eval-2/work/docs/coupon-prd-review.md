> **评审重点：** 需求完整性与逻辑闭环（遗漏识别），不纠结实现细节
> **项目背景：** 全新（greenfield）项目，本地除本 PRD 外无任何代码或表结构，可行性评估以行业通用实践为参照基线

## 评审综述

本 PRD 描述了优惠券的"快乐路径"（领取 → 下单抵扣），但**全文未覆盖任何异常分支、边界条件、并发场景与状态流转**。对于优惠券这类直接关联资金/营销预算的功能，缺失的部分恰恰是事故高发区（超发、重复使用、抵扣金额错误、退款时券归属错乱）。下文按三个维度列出遗漏，并附可直接转发给 PM 的澄清清单。

---

## 1. 逻辑漏洞与边界缺失

### [P0-阻断] 折扣券缺少"封顶"与抵扣计算口径，金额计算无法闭环
- **问题描述：** PRD §2.1 定义"折扣券：8 折"，§2.3 仅说"优惠券金额从订单总额中扣除"。但折扣券是按比例抵扣，没有固定金额；同时未说明：① 是否有最高优惠封顶（如"最多减 50 元"），② 折扣作用于"商品总额"还是"含运费/税费的应付总额"，③ 抵扣后金额出现小数时如何取整。无这些规则，开发无法实现一个确定的抵扣金额。
- **潜在影响：** 大额订单使用折扣券可能产生远超营销预算的优惠；不同实现对"总额"理解不一致会导致线上抵扣金额与用户预期/财务对账不符。
- **引用：** PRD §2.1、§2.3

### [P0-阻断] 满减券的"满"判定基准未定义，使用门槛逻辑缺失
- **问题描述：** §2.1"满 100 减 20"未说明"满 100"指的是**订单商品总额**还是**应付总额**、是否**含运费**、是否在**叠加其他优惠后**仍需满足门槛、以及部分商品不参与活动时如何计算门槛。门槛判定是满减券的核心逻辑，目前完全缺失。
- **潜在影响：** 门槛判定口径不明会直接导致"该用的券用不了 / 不该用的券用了"，引发用户投诉与资损。
- **引用：** PRD §2.1、§2.3

### [P0-阻断] 优惠券领取与使用的并发/幂等控制完全缺失（超发与重复使用）
- **问题描述：** §2.2 领取、§2.3 使用均未提及任何并发与防重控制。优惠券是典型的高并发争抢场景，行业普遍存在两类事故：① **领取超发**——多请求并发通过"剩余库存校验"导致发放量超过预算；② **重复使用/一券多单**——同一张券在并发下单或重试中被多次核销抵扣。PRD 未约束库存总量、未约束每用户领取上限、未定义券的核销状态机，无法防御上述问题。
- **潜在影响：** 直接资损（超发的券都是真金白银），并可能被羊毛党批量套利。
- **引用：** PRD §2.2、§2.3；行业实践见 [race condition in coupon redemption](https://blogs.jsmon.sh/what-is-race-condition-in-coupon-redemption-ways-to-exploit-examples-and-impact/)、[腾讯云·优惠券系统设计](https://cloud.tencent.com/developer/article/2465372)

### [P0-阻断] 优惠券核销状态机缺失，下单失败/取消/退款后券的归属未定义
- **问题描述：** PRD 只描述"下单时抵扣"，但未定义券在整个订单生命周期中的状态流转：下单但**未支付**时券是否锁定？**支付失败 / 用户取消订单 / 超时关单**时券是否**退还**？**售后退款（整单退 / 部分退）**后券如何处理、是否返还、返还后有效期如何计算？这是一个典型的状态机，目前完全是空白。
- **潜在影响：** 用户可能"用了券但订单没成交，券却消失"（投诉、资损客户侧）；也可能"订单退款后券被退还又能再用"造成一单多用（资损平台侧）。这是优惠券最常见的资损与客诉来源。
- **引用：** PRD §2.3、§4；行业实践见 [reservation pattern 避免券抵扣竞态](https://medium.com/@mmdGhanbari/avoid-race-conditions-with-reservation-pattern-bc4846602417)

### [P1-需确认] 多券叠加 / 互斥规则未定义
- **问题描述：** §2.3 写"下单时选择一张优惠券"，似乎暗示单张，但未明确：① 是否允许满减券+折扣券**叠加**使用，② 与平台其他营销（如限时折扣、会员价、积分抵扣）是否**互斥**，③ 同一商品/订单上多种优惠的**计算先后顺序**。"一张"是字面限制还是仅当前阶段简化，需 PM 明确。
- **潜在影响：** 叠加规则不清会导致优惠计算结果不可预期，且后续接入新营销活动时返工。
- **引用：** PRD §2.3；行业实践见 [促销活动叠加互斥概念模型](https://blog.csdn.net/vincent1007/article/details/144904435)

### [P1-需确认] 优惠券适用范围（可用商品/品类/门店）未定义
- **问题描述：** PRD 未说明券是**全场通用**还是限定**指定商品 / 品类 / 店铺 / 渠道**。这直接决定门槛与抵扣的计算对象，也影响"下单页展示可用优惠券"的筛选逻辑（§3）。
- **潜在影响：** 范围规则缺失会使"可用券"判定无据可依，前端无法正确过滤，财务也无法做营销归因。
- **引用：** PRD §2.1、§2.3、§3

### [P1-需确认] 有效期规则不完整（生效时间、领取后 vs 固定区间、过期临界）
- **问题描述：** §2.2 仅说"每张优惠券有有效期，过期后自动失效"。未定义：① 有效期是**固定起止日期**还是**领取后 N 天**，② 是否有**生效时间**（领取后立即可用还是延迟生效），③ 过期判定的**时区与临界时刻**（精确到秒/日），④ 下单时券有效、但支付时已过期如何处理。
- **潜在影响：** 时区/临界处理不当会导致"提前失效或延迟失效"，引发客诉与资损。
- **引用：** PRD §2.2

### [P2-优化] 退款/失败场景的用户可见性与提示流程未描述
- **问题描述：** 当券因订单取消被退回、或下单时券已被占用/失效，PRD 未描述用户应被告知什么、能做什么操作（重选券 / 提示已退回我的优惠券）。
- **潜在影响：** 体验问题，不阻塞主流程，但建议在交互设计阶段补齐。
- **引用：** PRD §2.3、§3

## 2. 技术可行性与架构风险

> 说明：本项目为全新项目，无既有架构可评估改造成本。以下风险基于"该功能要落地必须具备的能力"与行业通用实践给出，供 PM/架构在立项时纳入考量，**不替开发做技术选型**。

### [P0-阻断] 缺少防超发/防重复使用的一致性保障要求（需求侧未给出约束）
- **技术分析：** 防超发与防重复核销在工程上需要原子的"校验+扣减"与幂等控制。但这要求 PRD 先在**需求层面**给出约束：库存总量、每用户领取上限、单券是否一次性核销等。当前 PRD 未提供任何此类约束，导致一致性方案无的放矢。
- **改进建议（方向性，最终由开发选型）：**
  - 方案 A：需求侧明确"库存上限 + 每人限领 + 一券一次性核销"，工程侧用原子扣减/分布式锁/幂等键实现。
  - 方案 B：若本期为小流量内测，可接受较弱保障，但需 PM 显式确认预期并发量级与可接受的超发风险。
- **引用：** PRD §2.2、§2.3；行业实践见 [腾讯云·优惠券系统设计](https://cloud.tencent.com/developer/article/2465372)、[coupon abuse 2026 指南](https://devsecopsschool.com/blog/coupon-abuse/)

### [P1-需确认] 预期规模未量化，无法评估方案与容量
- **技术分析：** PRD 未给出预期用户量、活动并发峰值、优惠券发放量级、单活动库存量。优惠券常伴随营销活动产生瞬时高并发（抢券），规模直接决定架构复杂度（是否需要缓存预热、限流、异步发券）。
- **改进建议：** 请 PM 提供预期量级，作为后续技术方案的输入；无需在 PRD 中确定具体技术。
- **引用：** PRD §1、§2.2

### [P2-优化] 风控/反薅羊毛能力未提及
- **技术分析：** PRD 未涉及对批量注册、机器领取、同人多账号的风控约束。营销券是羊毛党重点目标。
- **改进建议：** 至少在需求层面声明是否需要领取频率限制、设备/账号风控，作为是否接入风控系统的依据。
- **引用：** PRD §2.2；行业实践见 [coupon abuse 2026 指南](https://devsecopsschool.com/blog/coupon-abuse/)

## 3. 需与 PM 二次确认的疑问清单 (Todo List)

> 以下问题可直接转发给 PM，无需二次编辑。

- [ ] **关于折扣券抵扣口径：** "8 折"是否有**最高优惠封顶**（如最多减 X 元）？折扣作用于**商品总额**还是**含运费/税的应付总额**？抵扣出现小数时如何取整（向下取整/四舍五入）？
- [ ] **关于满减券门槛：** "满 100"的判定基准是**商品总额**还是**应付总额**？**是否含运费**？是否在**叠加其他优惠后**仍需满足门槛？存在不参与活动的商品时如何计算门槛？
- [ ] **关于发放量与领取上限：** 每个活动/每种券的**库存总量**是多少？**每个用户**最多能领几张同种券？是否需要"先到先得、抢完为止"？
- [ ] **关于订单生命周期中的券状态：** 下单未支付时券是否**锁定**？**支付失败/取消/超时关单**时券是否**退还**且有效期如何处理？**售后退款（整单/部分）**后券是否返还、能否再次使用？
- [ ] **关于叠加与互斥：** 是否允许**多张券叠加**？与平台其他营销（限时折扣、会员价、积分抵扣）是否**互斥**？多种优惠的**计算顺序**是什么？"选择一张"是最终规则还是本期简化？
- [ ] **关于适用范围：** 券是**全场通用**还是限定**指定商品/品类/店铺/渠道**？这决定门槛与抵扣的计算对象，以及下单页"可用券"的筛选逻辑。
- [ ] **关于有效期：** 有效期是**固定起止日期**还是**领取后 N 天**？是否有**生效延迟**？过期判定的**时区与临界时刻**如何定义？下单时有效、支付时过期如何处理？
- [ ] **关于预期规模：** 预期的**用户量、抢券并发峰值、单活动发放量级**大约是多少？是否预期出现瞬时抢券高峰？（用于评估方案与容量，不需指定技术）
- [ ] **关于风控：** 是否需要对**机器领取/同人多账号/批量注册**做限制？是否有领取频率限制要求？
- [ ] **关于多端一致性：** §3 中"活动页可领列表""我的优惠券""下单页可用券"三处的状态以何为准？是否要求实时一致（如刚领取立即可见、刚使用立即置灰）？
