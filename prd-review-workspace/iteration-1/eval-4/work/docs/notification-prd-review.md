## 1. 逻辑漏洞与边界缺失

### [P0-阻断] "群发全部用户" 的扇出（fan-out）逻辑与一致性完全缺失
- **问题描述：** PRD §3 要求"大促期间需支持向全部用户群发活动通知"，但全程未描述一条逻辑消息如何扇出为 N 个用户 × M 个渠道的投递任务。群发是同步还是异步？是否落库为可追溯的投递任务？群发过程中失败/超时的用户如何补偿？这是通知系统最核心也最容易被低估的逻辑（一条逻辑消息→百万级设备 token 的映射是真正的瓶颈）。
- **潜在影响：** 若按同步实现，大促群发会瞬间打爆 DB 写入与下游渠道连接，导致整体不可用；若无投递任务落库，则无法回答"哪些用户没收到、为什么没收到"，验收标准"各类通知能正确推送"无法度量。
- **引用：** PRD §3 性能要求、§4 验收标准

### [P0-阻断] 用户"关闭某类通知"与"系统/安全类通知"的优先级冲突未定义
- **问题描述：** PRD §2.3 称"用户可在设置页开启/关闭某类通知"，§4 验收标准要求"用户关闭某类通知后不再收到该类通知"。但 §2.1 的系统通知包含"账号安全"。若用户关闭了系统通知，账号安全告警（如异地登录、改密）是否还发送？强制类/合规类通知是否应豁免于用户开关之外？PRD 未对"可关闭通知"与"强制通知"做区分。
- **潜在影响：** 字面执行验收标准会导致安全告警被用户静默关闭，引发安全与合规风险；若不执行又违背验收标准，逻辑自相矛盾，开发无所适从。
- **引用：** PRD §2.1（系统通知-账号安全）、§2.3、§4

### [P0-阻断] 通知触发源（event）与通知中心的关系未闭环
- **问题描述：** PRD 只列出了通知"类型"，但未说明通知由谁触发、何时触发。例如"发货通知"由订单/物流系统发出还是通知中心轮询？通知中心是被动接收上游事件，还是主动查询业务库？触发的接口契约、事件来源未定义，整条业务流的入口是断的。
- **潜在影响：** 缺少触发契约会导致通知中心与业务系统职责边界不清，无法开始接口设计；也无法评估事件丢失/重复时的处理。
- **引用：** PRD §2.1（订单/活动/系统通知）

### [P1-需确认] 通知去重与幂等缺失
- **问题描述：** PRD 未提及同一业务事件重复触发（上游重试、消息重复消费）时如何去重。行业通用做法是用 `event_id`/幂等键 + `user_id` 在投递前去重，PRD 无任何约定。
- **潜在影响：** 上游重试或队列 at-least-once 语义下，用户会收到重复通知（如重复短信，且短信有真实成本），影响体验并产生费用。
- **引用：** PRD §2（功能描述整体）

### [P1-需确认] 多渠道投递的失败、降级与回退逻辑缺失
- **问题描述：** PRD §2.2 列出站内信/短信/App Push 三渠道，但未说明：单条通知是"全渠道都发"还是"按优先级择一"？某渠道（如短信网关）失败时是否回退到其他渠道？是否重试？重试几次？App Push 的 token 失效如何处理？
- **潜在影响：** 渠道故障时通知静默丢失，用户该收到的订单/安全通知未达，且无降级路径。
- **引用：** PRD §2.2

### [P1-需确认] 通知状态机与"已读/未读/已送达"语义缺失
- **问题描述：** 站内信通常需要"未读/已读"状态及未读数；PRD 未定义站内信是否需要已读状态、是否有保留期与清理策略、是否需要送达回执（webhook 闭环）。三类渠道的"送达"定义不同（站内信落库即达？短信以网关回执为准？Push 以平台回执为准？），PRD 未统一。
- **潜在影响：** 验收标准"正确推送到对应渠道"无可度量的"送达"定义，验收会产生歧义。
- **引用：** PRD §2.2、§4

### [P2-优化] 用户通知设置的默认值与粒度未明确
- **问题描述：** §2.3 仅说"开启/关闭某类通知"。开关粒度是"按类型"（订单/活动/系统）还是可细到"按渠道 × 按类型"？新用户默认全开还是部分开？这些影响交互与数据模型，但不阻塞主流程。
- **潜在影响：** 粒度选择影响后续可扩展性，宜尽早确定，但可在设计阶段补充。
- **引用：** PRD §2.3

## 2. 技术可行性与架构风险

### [P0-阻断] "实时到达" + "全员群发" 两项要求存在内在张力，且无量化指标
- **技术分析：** PRD §3 同时要求"全部用户群发"与"实时到达"。行业经验表明这两者在大规模下是矛盾的：例如向千万级设备投递，受 APNs/FCM 平台限流、短信网关 TPS 限制约束，本质上必须排队、分批、限流，无法对全员做到严格"实时"。当前为空白工程，无任何队列、worker 池、限流组件，要从零支撑"大促全员群发"需要完整的异步管线（消息队列 + 优先级队列 + 分渠道 worker 池 + 限流/背压 + 死信队列）。
- **改造成本：** 从零搭建异步通知管线（消息队列、分渠道 dispatcher、幂等存储、DLQ、限流），属于一个独立子系统的体量，远超 PRD 当前篇幅所暗示的复杂度。
- **改进建议：**
  - 方案 A：将"实时"按通知优先级分级——安全/订单类高优先级走快通道尽量实时；活动群发类标记为"可批量、可延迟"，明确允许在一个时间窗口（如分钟级）内完成投递。优点：符合行业实践、可落地；缺点：需 PM 接受"群发非严格实时"。
  - 方案 B：坚持全员严格实时——需投入大量基础设施且仍受第三方平台限流硬约束，性价比低、风险高。
  - （注：具体队列/中间件选型属实现细节，由开发在设计阶段决定，此处不展开。）
- **引用：** PRD §3；行业实践见文末来源

### [P1-需确认] 大促群发对下游与数据库的冲击（背压/限流）未在需求层面被认知
- **技术分析：** 全员群发的真正瓶颈是扇出导致的海量 DB 写入与下游 API 调用，易触发数据库写吞吐打满、第三方网关限流。PRD 未提出任何限流、分批、背压要求，意味着需求方未意识到此约束，可能在设计/压测阶段才暴露。
- **改造成本：** 需在管线中引入分渠道限流与背压机制（隔离 push/短信/站内信 worker 池，避免一个渠道拖垮其他）。
- **改进建议：** 在需求中明确"群发允许限流分批投递"，并要求 PM 提供量化指标（见 Todo）。
- **引用：** PRD §3；行业实践见文末来源

### [P2-优化] 短信渠道的成本与合规（频控）未纳入考量
- **技术分析：** 短信为按条计费且受运营商频控/合规约束。活动类通知全员群发短信成本高且可能触发运营商风控。PRD 未区分哪些类型走短信、是否有发送频次上限。
- **改进建议：** 建议活动类默认不走短信或设频控阈值；此为优化项，不阻塞主流程。
- **引用：** PRD §2.1（活动通知）、§2.2（短信）

## 3. 需与 PM 二次确认的疑问清单 (Todo List)

> 以下问题可直接转发给 PM，无需二次编辑。

- [ ] **关于性能指标（§3）：** "全部用户"的量级大约是多少（万级 / 百万级 / 千万级）？大促单次群发的目标用户规模峰值是多少？
- [ ] **关于"实时"定义（§3）：** "实时到达"的可接受时延上限是多少？是否可以接受活动群发类通知在一个时间窗口内（如 1～5 分钟）分批完成，而仅对安全/订单类要求秒级？
- [ ] **关于强制通知（§2.1 / §2.3 / §4）：** 用户关闭"系统通知"后，账号安全类告警（异地登录、改密等）是否仍需强制送达？哪些通知类型属于"用户不可关闭"的强制通知？
- [ ] **关于通知触发源（§2.1）：** 各类通知由哪个上游系统/事件触发（如订单系统发出"发货事件"）？通知中心是被动接收事件，还是主动查询业务数据？事件来源与触发时机请明确。
- [ ] **关于多渠道策略（§2.2）：** 同一条通知是"多渠道同时发"还是"按优先级择一发"？某渠道失败时是否回退到其他渠道？是否需要重试，重试几次？
- [ ] **关于去重规则（§2）：** 同一业务事件被上游重复触发时，是否需要去重？以什么作为去重键（如业务事件 ID + 用户 ID）？
- [ ] **关于"送达"的验收口径（§4）：** "正确推送到对应渠道"如何判定送达？站内信以落库为准、短信/Push 是否需要以网关/平台回执为准？是否需要送达回执与失败可观测？
- [ ] **关于站内信状态（§2.2）：** 站内信是否需要"已读/未读"与未读数？通知的保留时长与清理策略是怎样的？
- [ ] **关于设置粒度（§2.3）：** 开关是"按类型"还是"按类型 × 按渠道"？新用户的默认开关状态是什么？
- [ ] **关于短信成本/频控（§2.1 / §2.2）：** 活动类通知是否允许走短信群发？是否有单用户/全局的发送频次上限？
