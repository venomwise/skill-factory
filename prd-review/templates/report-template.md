# 评审报告模板

风险级别全文统一使用以下三档：

- `[P0-阻断]` — 高危，逻辑/架构存在阻断性缺陷，需 PM 或架构澄清后才能继续
- `[P1-需确认]` — 中危，需与 PM/团队确认，可在设计阶段处理但有风险
- `[P2-优化]` — 低危，优化建议，不阻塞推进

## 首次分析模板

Write `<prd 同级目录>/<prd-basename>-review.md`，使用以下结构：

```markdown
# PRD 需求评审报告

> **评审时间：** YYYY-MM-DD
> **PRD 文档：** [文件名](相对路径)
> **评审状态：** 🔴 待澄清 / 🟡 部分确认 / 🟢 已通过

---

## 1. 逻辑漏洞与边界缺失

### [P0-阻断] 风险点标题
- **问题描述：** 基于 PRD 第 X 章节/第 Y 段，缺少对...情况的处理。具体来说...
- **潜在影响：** 可能导致系统出现...异常，引发数据不一致/用户无法操作/...
- **引用：** PRD §章节名 或具体段落

### [P1-需确认] 风险点标题
- **问题描述：** ...
- **潜在影响：** ...
- **引用：** ...

---

## 2. 技术可行性与架构风险

### [P0-阻断] 风险点标题
- **技术分析：** 当前架构为...，PRD 要求...，存在以下问题：...
- **改造成本：** 预估需要...（时间/人力/新增组件）
- **改进建议：**
  - 方案 A：...（优点...，缺点...）
  - 方案 B：...（优点...，缺点...）
- **引用：** 当前项目 文件路径:行号 / PRD §章节名

---

## 3. 需与 PM 二次确认的疑问清单 (Todo List)

> 以下问题可直接转发给 PM，无需二次编辑。

- [ ] **关于 [具体功能模块]：** PRD 中提到"..."，请问这里的具体量化指标是什么？（如：响应时间 < ? ms，并发量 > ? QPS）
- [ ] **关于 [业务规则]：** "..."的判断标准是什么？是否有优先级排序规则？
- [ ] **关于 [异常处理]：** 当...失败时，用户可以做什么操作？系统如何恢复？
- [ ] **关于 [权限控制]：** 哪些角色可以执行...操作？是否需要审批流程？

---

## 4. 评审结论

- **[P0-阻断]：** X 个
- **[P1-需确认]：** Y 个
- **[P2-优化]：** Z 个
- **待 PM 澄清：** N 个问题

**下一步建议：**
- 若存在 P0 问题或大量待澄清项 → 等待 PM 回答后重新评审
- 若仅有 P1/P2 问题 → 可在设计阶段处理，建议进入技术方案设计
- 若已全部解决 → 需求澄清完成，可以开始设计技术方案
```

## 迭代分析

Edit `<prd 同级目录>/<prd-basename>-review.md`：

1. **更新评审状态：** 🔴 待澄清 → 🟡 部分确认 → 🟢 已通过
2. **标记已解决的问题：**
   ```markdown
   ### [已解决] ~~风险点标题~~
   - **原问题：** ...
   - **PM 确认：** （时间）PM 回复："..."
   - **解决方案：** 采用...方案
   ```
3. **新增发现的问题：** 基于 PRD 更新或 PM 补充的信息
4. **删除不再适用的问题：** 用删除线标记或直接移除
5. **添加变更记录：** 在报告末尾或开头增加
   ```markdown
   ## 变更记录
   - **YYYY-MM-DD：** 首次评审，发现 X 个高危问题
   - **YYYY-MM-DD：** PM 确认了问题 1、2、3，新增问题 4
   - **YYYY-MM-DD：** 所有问题已解决，评审通过
   ```
